3.  **Архитектура парсера (Железобетонный скрапер):**
    *   **Многоуровневый подход:**
        *   **Уровень 1 (Direct API):** Прямые вызовы внутренних API (предпочтительно), обнаруженных с помощью браузера.
        *   **Уровень 2 (Browser Network Discovery):** Если API недоступны, используется Playwright для открытия страниц, перехвата сетевых запросов (XHR/fetch) и автоматического обнаружения новых эндпоинтов.
        *   **Уровень 3 (DOM Extraction):** Резервный механизм парсинга HTML-структуры, если первые два уровня не сработали.
    *   **Самовосстановление:** Система должна автоматически переключаться на следующий уровень при сбое предыдущего и самостоятельно обновлять реестр рабочих эндпоинтов.
    *   **Разделение процессов:** Сбор данных (Scrape), хранение сырых данных (Store Raw JSON) и их обработка (Parse) должны быть разделены.
4.  **Инфраструктура:**
    *   **Планировщик задач (Scheduler):** Для запуска периодических задач сбора данных (ежедневный полный синк рынка, часовой синк аналитики и т.д.).
    *   **Очередь задач и воркеры:** Система должна использовать очередь (Redis) и несколько параллельных воркеров для выполнения задач скрейпинга, обеспечивая масштабируемость и отказоустойчивость.
    *   **Управление прокси:** Система failover для прокси (2-3 статических прокси), управляемая через API, без ротации на каждый запрос.
    *   **Stealth-режим для браузера:** Применение техник для маскировки автоматизации (отключение , имитация поведения человека).
5.  **Мониторинг:** API-эндпоинт для проверки состояния системы ().
